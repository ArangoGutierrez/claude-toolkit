# Workflow Library

Reusable, named agent-orchestration graphs shipped in
[`.claude/workflows/`](https://github.com/ArangoGutierrez/claude-toolkit/tree/main/.claude/workflows).
Each one is a deterministic DAG — fan-out, verification gates, and synthesis
encoded as a script, not as prompting habits. For the *why*, read
[Orchestration: Loops → Graphs](../orchestration.md).

## Requirements

- Claude Code **v2.1.154 or later** (dynamic workflows).
- Any paid plan; on **Pro**, enable the "Dynamic workflows" row in `/config`.
- Workflows in a project's `.claude/workflows/` are discovered automatically
  when the repo is cloned; copies in `~/.claude/workflows/` are available in
  every project (this repo's `deploy.sh` mirrors them there).
- Invocation: each workflow registers as a slash command named after its
  `meta.name` — type `/<name>` and describe inputs in natural language;
  Claude passes them to the script as structured `args`.

Source: [Claude Code workflows documentation](https://code.claude.com/docs/en/workflows).

## review-verify

The canonical find → verify graph: one finder agent per review dimension,
then one adversarial refuter per finding. A finding only reaches you after
surviving an agent whose sole job was to kill it.

```text
/review-verify                          # review the working diff, default dimensions
/review-verify the diff of feat-x vs main, dimensions: concurrency, error handling
```

- **Args:** `target` (what to review — diff spec, path, PR ref; default:
  working diff), `dimensions` (list; default: correctness, security,
  test quality).
- **Shape:** `pipeline(dimensions)` — verification of dimension A's findings
  starts while dimension B is still reviewing; no wasted wall-clock.
- **Cost:** `dimensions + findings` agents (default run with a clean-ish
  diff: 3–8 agents).

## chief-dispatch

The orchestrator pattern as a graph: a builder executes a fully-specified
brief, an adversarial critic verifies the work against the brief —
treating the builder's report as unverified claims — and a single bounded
fix round handles rejections. With more than one task, each builder gets an
isolated git worktree; the critic and fixer operate on the builder's
reported working directory and branch.

```text
/chief-dispatch the brief at docs/briefs/task-1.md
/chief-dispatch tasks: brief docs/briefs/task-1.md and brief docs/briefs/task-2.md
```

- **Args:** `tasks: [{brief}]` (file path or inline text) — or a single
  `brief`.
- **Returns:** per-task `DONE | DONE_WITH_CONCERNS | BLOCKED` with the
  branch each task's work sits on. **It never merges or pushes** — the
  operator reviews the returned branches and merges deliberately.
- **Cost:** 2–4 agents per task (builder, critic, optional fixer + re-gate).

## weekly-audit

A four-way parallel hygiene sweep — config audit, eval suite, docs drift,
dead references — synthesized into one prioritized report. The barrier
before synthesis is deliberate: the report needs all four sweeps.

```text
/weekly-audit
```

Runs headless-friendly: schedule it (cron, launchd, or Claude Code's
background tasks) to make a *loop launch a graph* — see the
[composition section](../orchestration.md#composing-the-layers) of the
orchestration guide. Note: MCP servers needing interactive auth may be
absent in headless runs; the auditors only use local tools.

- **Args:** none.
- **Cost:** 5 agents (4 sweeps + synthesis).

## Customizing

- **Models:** workflows inherit the session model. Pin a stage only when
  you are sure a cheaper tier fits (e.g. `effort: 'low'` for mechanical
  sweeps); when unsure, inherit.
- **Agent types:** stages accept `agentType` (e.g. a repo-specific
  `principal-engineer`) — the shipped defaults avoid it so the library
  works in any clone.
- **Budgets:** append a token target ("+300k") to scale depth; scripts can
  read it via the `budget` global.

## Writing your own

Start from a completed ad-hoc run: `/workflows` → select the run → press
`s` to save it (Tab toggles project vs personal scope). Keep `meta.name`,
`meta.description`, and `meta.whenToUse` single-line and single-quoted —
`scripts/gen-workflow-docs.sh` extracts them, and
`scripts/check-workflow-syntax.sh` gates the syntax in CI and locally.
